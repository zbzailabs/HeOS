# Human and AI Collaboration

HeOS 开发采用“人类负责方向与验收，AI 负责执行”的协作方式。本文档用于说明人类如何使用飞书项目、GitHub Projects 和 GitHub Issues 与 AI 配合推进开发。

## 1. 分工原则

人类负责：

- 提出需求、大方向和业务约束。
- 判断优先级。
- 发现问题并反馈。
- 验收功能、页面、流程和交付结果。
- 对业务边界、标准取舍和上线节奏作决策。

AI 负责：

- 将需求拆解为规格、任务和验收标准。
- 创建或补齐 GitHub Issues。
- 将任务加入 GitHub Projects。
- 编写代码、文档、测试和验证命令。
- 提交代码、关联 PR、更新任务状态。
- 在任务结束时同步飞书项目记录。

## 2. 工具职责

GitHub Issues 是开发任务唯一主源。每个可执行任务都必须有 Issue，Issue 记录需求、边界、验收标准、相关文件、验证命令和完成结果。

GitHub Projects 是开发看板和排期工具。它负责展示 Issue 状态、优先级、里程碑和执行进度。

飞书项目是管理汇报与外部协同记录。它负责记录任务启动、阶段进展、完成内容、验证结果、遗留项和流程节点。

## 3. 人类日常使用方式

人类主要通过自然语言提出需求，不需要维护完整技术细节。

常用指令示例：

```text
继续做 #4
```

```text
优先做智慧农业标准稽核页
```

```text
这个方向不对，设备数据要先围绕腾龙小学智慧农场做
```

```text
把这个问题建成 issue：告警关闭后再次超限要重新触发
```

```text
看一下当前 HeOS 项目进度，告诉我下一步做什么
```

AI 收到指令后负责判断是否已有 Issue。没有 Issue 时创建 Issue；已有 Issue 时补齐规格、验收标准和项目看板状态。

## 4. 开发流程

新任务开始：

- 先读取飞书项目现有管理内容。
- 确认是否已有 GitHub Issue。
- 若无 Issue，创建 Issue 并补齐验收标准。
- 将 Issue 加入 GitHub Projects。
- 明确是否采用 Spec-Driven Development；涉及开发和代码实现时，先写规格再写代码。

开发执行：

- 围绕 GitHub Issue 开发。
- commit、PR、验证结果必须关联对应 Issue。
- 状态流转在 GitHub Projects 中维护。
- 重大方向、业务边界或标准取舍需要人类确认。

任务结束：

- 汇总修改文件、完成内容、验证命令和结果。
- 说明未完成项和后续建议。
- 将 Issue 链接、PR 链接、完成内容、验证结果同步到飞书项目。
- 更新飞书项目流程节点。

## 5. 验收方式

人类验收时主要关注：

- 功能方向是否正确。
- 页面和流程是否符合实际业务。
- 是否符合农业农村部智慧农业标准体系建设指南的方向。
- 是否适配腾龙小学智慧农场等真实场景。
- 是否存在需要调整的业务判断。

AI 交付时必须说明：

- 做了什么。
- 改了哪些文件。
- 关联哪个 Issue。
- 运行了哪些验证。
- 哪些内容未完成。
- 哪些问题需要人类决策。

## 6. 推荐节奏

每次迭代先从 GitHub Projects 选择 1 到 3 个 Issue 进入开发。P0 任务优先于 P1，P1 任务优先于 P2。

人类只需要调整优先级和验收结果。AI 负责维护 Issue、Projects、PR、验证记录和飞书同步。

固定协作模型为：

```text
人类：方向、需求、问题、验收
AI：拆解、实现、验证、提交、记录
```
